*Team technical report. We take a small Python function, give the fragment it
uses a minimal executable **K** semantics ("mini Python"), state the function's
correctness as a **reachability rule** (a K `claim`), explain the proof
informally, and then give the entire machine-oriented formal proof.*

**Background in one paragraph.** [K](https://kframework.org)
([runtimeverification/k](https://github.com/runtimeverification/k)) is a
rewrite-based framework for *executable* formal semantics: you write a syntax, a
*configuration* of state cells, and rewrite *rules*, and K derives an
interpreter (`krun`), a compiler (`kompile`), and a reachability-logic prover
(`kprove`). K's logical foundation is **matching logic** (Roșu, *Matching
Logic*, LMCS 2017; Chen & Roșu, *Matching μ-Logic*, LICS 2019), and program
correctness is **reachability logic** (Roșu & Ștefănescu, FM 2012 / LICS 2013),
whose **Circularity** rule lets a loop's own specification serve as the
coinductive hypothesis for its proof — replacing the classical loop invariant.
This example is canonical: K Tutorial **Lesson 1.22** verifies exactly this
`sum` function, and our semantics and claims follow it.

**What is proved.** *Partial correctness*: if and when it terminates,
`sum(N) = N·(N+1)/2` for every `N ≥ 0`. **What is not:** termination (no variant
is given), and the K toolchain was **not executed in this environment** — the
artifacts are constructed to be `kompile`/`kprove`-checkable and mirror the
proven Lesson 1.22, and the proof is a faithful transcription of a hand
reachability proof that was independently adversarially reviewed (5 lenses, no
critical/major defects) and stress-tested (20k randomized loop checks,
`N = 0..11` end-to-end). See §5.5 for the full residual-risk accounting.

## 1. The program and its (informal) specification

`summation.py`:

```python
def sum(n):
    s = 0
    i = 1
    while i <= n:
        s += i
        i += 1
    return s
```

**Informal specification (as originally stated).** A function `sum` takes an
integer input `n` and calculates the sum of the numbers from one to `n`. It uses
a loop that goes from `1` all the way to `n` and sums up all the numbers into a
variable `s`, which is then returned.

**Formal reading.** For every integer `N ≥ 0`,

```
    sum(N)  =  1 + 2 + ... + N  =  N·(N+1)/2 .
```

The loop maintains `s = 1 + 2 + ... + (i−1)` while `i` walks from `1` to `N+1`;
on exit `i = N+1` and `s = N·(N+1)/2`. (For `N = 0` the loop body never runs and
`s = 0 = 0·1/2`.) The function uses only these constructs: integer literals and
names, `+`, `<=`, assignment `x = e`, augmented assignment `x += e`, a `while`
loop, a function `def`, a `return`, and a function call. Section 2 gives a
formal semantics for *exactly* these and nothing more.

## 2. The K semantics of the language ("mini Python")

We name the language **mini Python**. Its full semantics is the file
[mini-python.k](mini-python.k), reproduced below. A K definition is read as: a
*syntax* (BNF with evaluation-order annotations `strict`/`seqstrict`), a
*configuration* of cells, and *rewrite rules* `LHS => RHS` over those cells. The
`<k>` cell is the **computation list** (`~>` is its cons, `.K` the empty
computation); `seqstrict` makes K auto-generate *heating/cooling* rules that
evaluate sub-expressions left-to-right by pulling them to the front of `<k>`.

```k
module MINI-PYTHON-SYNTAX
  imports INT-SYNTAX
  imports BOOL-SYNTAX
  imports ID-SYNTAX

  // ---- expressions ----------------------------------------------------------
  syntax IExp ::= Id | Int
                | "(" IExp ")"            [bracket]
                > IExp "+" IExp           [seqstrict, left]   // arithmetic +
                | Id "(" IExps ")"        [strict(2)]         // function call

  syntax BExp ::= Bool
                | IExp "<=" IExp          [seqstrict]         // comparison <=

  // Argument/parameter lists. The shared klabel(exps) Bot/Bots construction makes
  // Ints (evaluated args, a KResult) and Ids (parameters) BOTH subsorts of the
  // strict IExps list: `IS:Ints` is a legal call argument, the seqstrict arg list
  // cools to an Ints KResult, and `.Bots` is the shared empty terminator usable as
  // both `.Ids` and `.Ints`.  (Cf. K Tutorial Lesson 1.22.)
  syntax Bot
  syntax Bots  ::= List{Bot,  ","}        [klabel(exps)]
  syntax Ints  ::= List{Int,  ","}        [klabel(exps)] | Bots
  syntax Ids   ::= List{Id,   ","}        [klabel(exps)] | Bots
  syntax IExps ::= List{IExp, ","}        [klabel(exps), seqstrict] | Ints | Ids

  syntax KResult ::= Int | Bool | Ints

  // ---- statements -----------------------------------------------------------
  // Suite-headed/leaf statements bind TIGHTER than sequencing (the `>` block),
  // and a Suite is itself a Stmt, so an INDENT...DEDENT block is self-delimiting.
  syntax Stmt ::= Id "=" IExp                       [strict(2)]   // assignment
                | Id "+=" IExp                                     // augmented
                | "while" BExp ":" Suite                          // while
                | "def" Id "(" Ids ")" ":" Suite                  // function def
                | "return" IExp                     [seqstrict]   // return
                | Id "(" IExps ")"                  [strict(2)]   // call stmt
                | Suite
                > Stmt Stmt                          [left]        // sequencing

  syntax Suite ::= "INDENT" Stmt "DEDENT"
endmodule

module MINI-PYTHON
  imports MINI-PYTHON-SYNTAX
  imports INT  imports BOOL  imports MAP  imports LIST

  configuration
    <k> $PGM:Stmt </k>
    <store> .Map  </store>     // program variables  Id |-> Int
    <funcs> .Map  </funcs>     // function table      Id |-> (def ...)
    <stack> .List </stack>     // call stack of saved (continuation, store)

  // expressions
  rule <k> X:Id => V ... </k>  <store> ... X |-> V ... </store>          // (lookup)
  rule <k> I1:Int + I2:Int  => I1 +Int  I2 ... </k>                      // (add)
  rule <k> I1:Int <= I2:Int => I1 <=Int I2 ... </k>                      // (leq)

  // statements
  rule <k> S1:Stmt S2:Stmt => S1 ~> S2 ... </k>                          // (seq)
  rule <k> INDENT S:Stmt DEDENT => S ... </k>                            // (suite)
  rule <k> X:Id = I:Int => .K ... </k>                                   // (asgn)
       <store> STORE => STORE [ X <- I ] </store>
  rule <k> X:Id += E:IExp => X = X + E ... </k>                          // (augasgn)

  // while: evaluate a FRESH copy of the guard; keep an UNEVALUATED copy frozen
  // in #whileLoop for re-checking. Re-entry is to `while`, never to an `if`.
  syntax KItem ::= #whileLoop(BExp, Suite)
  rule <k> while B:BExp : S:Suite => B ~> #whileLoop(B, S) ... </k>      // (while)
  rule <k> true  ~> #whileLoop(B:BExp, S:Suite) => S ~> while B : S ... </k>  // (while-t)
  rule <k> false ~> #whileLoop(_:BExp, _:Suite) => .K ... </k>          // (while-f)

  // functions
  rule <k> def FNAME:Id ( ARGS:Ids ) : BODY:Suite => .K ... </k>        // (def)
       <funcs> FS => FS [ FNAME <- def FNAME ( ARGS ) : BODY ] </funcs>
  rule <k> FNAME:Id ( IS:Ints ) ~> CONT => #makeBindings(ARGS, IS) ~> BODY </k>  // (call)
       <funcs> ... FNAME |-> def FNAME ( ARGS:Ids ) : BODY:Suite ... </funcs>
       <store> STORE => .Map </store>
       <stack> .List => ListItem(state(CONT, STORE)) ... </stack>
  rule <k> return I:Int ~> _ => I ~> CONT </k>                          // (return)
       <stack> ListItem(state(CONT, STORE)) => .List ... </stack>
       <store> _ => STORE </store>
  rule <k> return I:Int ~> .K => I </k>  <stack> .List </stack>         // (return-top)

  // parameter binding
  syntax KItem ::= #makeBindings(Ids, Ints)
                 | state(continuation: K, store: Map)
  rule <k> #makeBindings(.Ids, .Ints) => .K ... </k>
  rule <k> #makeBindings((I:Id, IDS:Ids) => IDS, (IN:Int, INTS:Ints) => INTS) ... </k>
       <store> STORE => STORE [ I <- IN ] </store>
endmodule
```

**Indentation modeling note.** K is not whitespace-sensitive, so it cannot apply
Python's off-side rule at parse time. A real Python tokenizer emits `INDENT`/
`DEDENT` tokens bracketing a suite; we model a suite *exactly* as that block,
`Suite ::= "INDENT" Stmt "DEDENT"`. Displayed indentation maps 1-to-1 to
`INDENT ... DEDENT`. This touches only lexing/layout, never the semantics.

**Rule-by-rule.** `(lookup)` reads a variable from `<store>`. `(add)`/`(leq)`
fire once their operands are `Int`s (their `seqstrict` heating evaluates the
operands first — these heating/cooling micro-steps *are* the manual
"lookup/add" steps of a paper proof). `(seq)` threads two statements onto `<k>`;
`(suite)` strips the layout brackets. `(asgn)` updates `<store>`; `(augasgn)`
desugars `x += e` to `x = x + e` (value-exact for the simple-name targets used
here). The **while triple** is the heart: `(while)` evaluates a fresh guard copy
while freezing an unevaluated copy in `#whileLoop`; `(while-t)` runs the body and
*rebuilds the `while`* (re-entry is to `while`, **never an `if`** — none is
introduced); `(while-f)` exits. The function rules `(def)`/`(call)`/`(return)`
keep a `<funcs>` table and a `<stack>` of `state(continuation, store)` frames:
`(call)` pushes the caller's frame and gives the callee a fresh `.Map` scope;
`(return)` pops the frame, restores the caller store, and delivers the value to
the saved continuation; `#makeBindings` binds parameters positionally.

This semantics covers **only** the constructs `sum` uses; there is deliberately
no `if`, no `None`, and an unbound read simply gets stuck (never exercised here).

## 3. The reachability rule — the specification of the function

The function's overall correctness is one **reachability rule** `φ_pre ⇒ φ_post`
between configuration patterns: starting from a configuration that defines `sum`
and calls it on a non-negative `N`, execution reaches a terminated configuration
in which `result` holds `N·(N+1)/2`. Abstractly:

```
  φ_pre  ≡  ⟨ def sum(n): <body>   result = sum(N) ⟩_k  ⟨ result ↦ _ ⟩_store
            ⟨ .Map ⟩_funcs  ⟨ .List ⟩_stack    ∧   N ≥ 0
  φ_post ≡  ⟨ · ⟩_k  ⟨ result ↦ N·(N+1)/2 ⟩_store  ⟨ ?_ ⟩_funcs  ⟨ .List ⟩_stack
```

In K this is the **(SUM)** `claim` in [mini-python-spec.k](mini-python-spec.k):

```k
claim
  <k>
    def sum ( n ) : INDENT
      s = 0
      i = 1
      while i <= n : INDENT s += i  i += 1 DEDENT
      return s
    DEDENT
    result = sum ( N:Int )
  => .K ... </k>
  <funcs> .Map => ?_:Map </funcs>
  <store> result |-> (_:Int => N *Int (N +Int 1) /Int 2) </store>
  <stack> .List </stack>
  requires N >=Int 0
  [all-path]
```

**How to read the claim.** A K claim is a reachability rule: each cell's
`X => Y` is a rewrite from pre- to post-state, conjoined. `requires N >=Int 0` is
the **precondition** (a matching-logic side constraint framed on every step).
The `<k>` cell rewrites the whole program to `.K` (terminated). The post-`<store>`
asserts `result |-> N*(N+1)/2`; `<funcs> .Map => ?_:Map` says a function table
now exists (the `?` marks an existential — *some* table, witnessed by the `sum`
entry); `<stack> .List` says the stack is balanced again after return. The
`[all-path]` attribute demands that *every* terminating path reach the target
(strictly stronger than one-path; sound here because mini Python is
deterministic — see §5.5). This is a **partial-correctness** statement: it
constrains terminating runs and asserts nothing about whether the loop halts.

## 4. Informal proof

### The loop's circularity

The whole proof turns on a single auxiliary claim about the loop — its **(LOOP)**
circularity, generalized over the accumulator `S` and the counter value `I`:

```
  (LOOP)   ⟨ while i<=n: (s+=i; i+=1) | s↦S, i↦I, n↦N ⟩  ∧  I ≤ N+1
       ⇒   ⟨ · | s ↦ S + (I+N)·(N−I+1)/2,  i ↦ N+1,  n ↦ N ⟩
```

In K (the **(LOOP)** claim in [mini-python-spec.k](mini-python-spec.k)):

```k
claim
  <k> while i <= n : INDENT s += i  i += 1 DEDENT => .K ... </k>
  <store>
    s |-> (S:Int => S +Int (I +Int N) *Int (N -Int I +Int 1) /Int 2)
    i |-> (I:Int => N +Int 1)
    n |-> N:Int
  </store>
  requires I <=Int N +Int 1
  [all-path]
```

It reads: *running the loop from counter `i = I` (with `I ≤ N+1`) adds
`Σ_{k=I}^{N} k = (I+N)·(N−I+1)/2` to `s` and leaves `i = N+1`.* This is the
arithmetic-series formula; at `I = 1` it gives the whole sum `N·(N+1)/2`.

### Why a circularity, and how the proof goes (in English)

Classically you'd prove a loop with a hand-chosen *invariant*. Reachability logic
replaces that with **coinduction**: K adds *every* claim in the module to its set
of hypotheses, so **(LOOP) is allowed to assume itself** while proving itself —
*provided* it first makes real progress (executes at least one genuine step
before reusing the hypothesis). That guardedness is exactly what keeps the
coinduction sound; the running-sum formula `S + (I+N)·(N−I+1)/2` plays the role
the invariant used to.

**Proving (LOOP).** Run the loop one step. The `(while)` rule evaluates the guard
`i <= n` — this is the genuine first step that earns the right to reuse the
hypothesis — and then we split on its value:

- **Guard true (`I ≤ N`):** the body runs. `s += i` makes `s = S + I`; `i += 1`
  makes `i = I + 1`. Control returns to the same `while`. Now we **invoke (LOOP)
  itself** at the shifted state `S := S+I, I := I+1` (its precondition
  `I+1 ≤ N+1` holds because `I ≤ N`). It tells us the rest of the loop adds
  `(I+1 .. N)`'s sum, leaving `s = (S+I) + Σ_{k=I+1}^{N} k`. The one arithmetic
  fact to check is that peeling off the first term is consistent:
  `I + Σ_{k=I+1}^{N} k = Σ_{k=I}^{N} k`. It is. ✓
- **Guard false (`I > N`, and `I ≤ N+1` from the precondition):** then `I = N+1`,
  the body never runs, `s` is unchanged, and the formula gives the empty sum `0`.
  So `s = S + 0 = S` and `i = N+1`. ✓

Both branches land on (LOOP)'s claimed post-state, and the hypothesis was used
only after a real step, so the circularity discharges. The side condition
`I ≤ N+1` is **necessary**: drop it and for `I ≥ N+2` the body still never runs
but the formula is non-zero, which would be false.

**Proving the function (SUM).** Now just execute the program against the
semantics: `def sum` files the function body into `<funcs>`; `result = sum(N)`
evaluates the argument, and `(call)` pushes the caller's frame, gives the callee
a fresh scope, and binds `n = N`. The body runs `s = 0`, `i = 1`, then the loop —
and here we **use (LOOP) as a lemma** at `S = 0, I = 1` (its precondition
`1 ≤ N+1` follows from `N ≥ 0`), which makes `s = 0 + N·(N+1)/2`. Then `return s`
pops the frame, restores the caller's store, and the returned value is assigned
to `result`. The final store is `result |-> N·(N+1)/2` — exactly the spec. ∎

That is the entire argument. Section 5 makes every step machine-precise.

## 5. The full formal proof (for machines)

This is the K-prover-checkable reachability proof, rebased from a prior hand
proof onto the **actual** mini Python rules. It presents both the `kprove` view
(how K's reachability prover discharges the claims) and the explicit
reachability-logic proof tree mapped onto the K rules.

### 5.0 Notation, configuration, closed form

Lowercase `s, i, n, sum, result` are program `Id`s; uppercase `S, I, N` are
matching-logic variables over `Int`; `?_` is an existential. `+Int, -Int, *Int,
/Int, <=Int, modInt` are the `INT` builtins (arbitrary precision, matching
Python `int`; `/Int` is integer division). Cells: `<k>` (computation list, `~>`
cons, `.K` empty), `<store>` (`Id |-> Int`), `<funcs>` (`Id |-> def...`),
`<stack>` (a `List` of `state(continuation, store)` frames). Abbreviations:
`W := while i <= n : INDENT s += i  i += 1 DEDENT`,
`Bdy := INDENT s += i  i += 1 DEDENT`.

**Running closed form (the form the (LOOP) claim uses):**

```
  cfA(I,N) := (I +Int N) *Int (N -Int I +Int 1) /Int 2  =  Σ_{k=I}^{N} k
            = (N*(N+1) - (I-1)*I)/2          [hand-proof "difference" form, equal]
  cfA(1,N) = N*(N+1)/2 .
```

### 5.1 Correspondence: hand-proof element ⇄ K rule / `kprove` mechanism

| Reachability-logic element | mini Python rule / `kprove` mechanism |
|---|---|
| `(lookup)` axiom | `(lookup)` rule, `X:Id => V` from `... X |-> V ...` |
| `(add)`/`(leq)` + operand micro-steps | `(add)`/`(leq)`; micro-steps **are** the auto-generated `seqstrict` heating/cooling for `+`/`<=` |
| `(seq)` | `(seq)` rule `S1 S2 => S1 ~> S2` |
| `(asgn)` | `(asgn)` rule `X = I:Int => .K` (`strict(2)` evaluates the RHS first) |
| `(while) ⇒ if {…} else {}` axiom | **replaced** by the while triple `(while)/(while-t)/(while-f)`; the guardedness-discharging first transition is `while ⇒ guard ~> #whileLoop` |
| `(cond-t)/(cond-f)` + `(Case Analysis)` | the prover's `#Or` split between `(while-t)` and `(while-f)` under `requires I <=Int N+1` |
| `(Circularity)` with guardedness | K uses the **(LOOP) claim itself** as the coinduction hypothesis (every claim is a circularity; cf. the `trusted` attribute) |
| `(Consequence)` → SMT oracle | Z3, plus the `[simplification]` lemmas (map-extensionality; exact halving) as the VC oracle |
| explicit `(Axiom)` framing of `<k>` tail, `n↦N`, side cond. | K's automatic `...` cell-completion |
| `∃S0,I0` + `(Abstraction)` of `φ_pre` | **no counterpart** — `(call)` gives a fresh `.Map`, so `s,i` are *written*, not overwritten |
| anything on an `if`-term | **does not carry over** — mini Python has no `if` |
| `=> .K` under `[all-path]` | an all-path reachability rewrite; sound here by determinism (§5.5) ⇒ all-path = one-path |

The matching-logic connectives `#And`/`#Or`/`#Equals`/`#Not`/`#Exists` are what
`kprove` manipulates: `#And` conjoins path conditions, `#Or` is the case-split,
and implication/equality goals are discharged by the oracle.

### 5.2 PART A — proof of (LOOP) by circularity

K adds (LOOP) to its circularities; we may reuse it **only after ≥ 1 genuine
rewrite** (guardedness). Target `RHS := ⟨ · | s↦ S +Int cfA(I,N), i↦ N+1, n↦ N ⟩`.

**A1 — the while-step (progress).** Rule `(while)`:

```
  ⟨ W | s↦S, i↦I, n↦N ⟩  =>  ⟨ (i<=n) ~> #whileLoop(i<=n, Bdy) | s↦S, i↦I, n↦N ⟩
```

This genuine `=>+` transition **discharges coinductive guardedness** — it
strictly precedes any reuse of (LOOP). The frozen `#whileLoop(i<=n, Bdy)` keeps
an unevaluated guard+suite; the head and frozen copies are distinct subterms, so
heating the head never touches the frozen one.

**A2 — evaluate the guard.** `<=` is `seqstrict`, so heating evaluates `i`
(`(lookup)` ⇒ `I`) then `n` (⇒ `N`); `(leq)` rewrites `I <= N => I <=Int N`. The
head is now `(I <=Int N) ~> #whileLoop(i<=n, Bdy)`, with `n↦N` and `I <=Int N+1`
framed by `...`.

**A3 — case split (`(while-t)` vs `(while-f)`).** This `#Or` on `(I <=Int N)`
under `I <=Int N+1` is the hand `(cond-*)` + `(Case Analysis)`.

- **CASE TRUE `(I <=Int N)` — `(while-t)`:**
  `true ~> #whileLoop(i<=n, Bdy) => Bdy ~> while i<=n : Bdy` (the `while` is
  rebuilt, not a fresh `if`). The body runs:
  1. `(suite)`: `INDENT s+=i  i+=1 DEDENT => s+=i  i+=1`; `(seq)` ⇒ `s+=i ~> i+=1`.
  2. `s += i` `(augasgn)`→ `s = s + i`; RHS `seqstrict`: `(lookup)` `S, I`; `(add)`
     `S +Int I`; `(asgn)` ⇒ `s ↦ S +Int I`.
  3. `i += 1` `(augasgn)`→ `i = i + 1`: `(lookup)` `I`; `(add)` `I +Int 1`;
     `(asgn)` ⇒ `i ↦ I +Int 1`.
  State: `⟨ while i<=n : Bdy | s↦ S+Int I, i↦ I+Int 1, n↦ N ⟩ ∧ I <=Int N`.
  **Reuse (LOOP)** at `{S := S+Int I, I := I+Int 1}`; precondition `(I+1) <=Int
  N+1` follows from `I <=Int N` (Z3). It rewrites `while... => .K` and updates
  `s ↦ (S+Int I) +Int cfA(I+1,N)`, `i ↦ N+1`. By **VC1**,
  `(S+Int I) +Int cfA(I+1,N) = S +Int cfA(I,N)` ⇒ `RHS`. Used strictly after A1. ✓
- **CASE FALSE `(I >Int N) ∧ (I <=Int N+1)` — `(while-f)`:** antisymmetry ⇒
  `I = N+1`; `false ~> #whileLoop(_,_) => .K`, store unchanged: `s↦S`,
  `i↦ I = N+1`. By **VC2**, `cfA(I,N) = 0`, so `s = S +Int 0 = S +Int cfA(I,N)`
  and `i = N+1` ⇒ `RHS`. Circularity not used. ✓

`(Case Analysis)` (`#Or`) + `(Transitivity)` over A1's progress give
`A ∪ {(LOOP)} ⊢ (LOOP)` with guardedness met, so the circularity discharges:
`A ⊢ (LOOP)`. ∎

### 5.3 PART B — proof of (SUM) (the real def/call/return layer)

Unlike a bare IMP proof, this executes the full call machinery.

- **B1 — `def`.** `(def)`: `def sum(n) : BODY => .K`, with
  `<funcs> .Map => sum |-> def sum(n) : BODY`,
  `BODY = INDENT s=0  i=1  W  return s DEDENT`. (`(seq)` first splits
  `def... ~> (result = sum(N))`.) This entry witnesses the post-store `?_:Map`.
- **B2 — argument eval + call.** `result = sum(N)` is `strict(2)`, so RHS
  `sum(N)` heats; the call is `strict(2)` over the `seqstrict` `IExps` list;
  `N` is already `Int`, so the arg list cools to `(N) : Ints` (a `KResult`). The
  cooled continuation `CONT = (result = ☐)` waits. `(call)` fires: **pushes**
  `state(CONT, STORE)` onto `<stack>`, resets `<store>` to `.Map`, queues
  `#makeBindings((n),(N)) ~> BODY`. (Fresh frame ⇒ the hand `∃S0,I0`+
  `(Abstraction)` has *no counterpart* — `s,i` are first written.)
- **B3 — `#makeBindings`.** `#makeBindings((n),(N)) => #makeBindings(.Ids,.Ints)`
  with `<store> .Map => n |-> N`, then `=> .K`.
- **B4 — body init.** `BODY` `(suite)`-unwraps and `(seq)`-sequences to
  `s=0 ~> i=1 ~> W ~> return s`. `(asgn)`: `s ↦ 0`, then `i ↦ 1`. Store:
  `s↦0, i↦1, n↦N`.
- **B5 — the loop via (LOOP) as a lemma.** The state matches (LOOP)'s LHS at
  `{S := 0, I := 1}`; precondition `1 <=Int N+1` follows from `N >=Int 0` (Z3).
  Applying (LOOP) rewrites `W => .K` and updates `s ↦ 0 +Int cfA(1,N)`, `i ↦ N+1`.
  By **VC3**, `cfA(1,N) = N*(N+1)/2`, so `s ↦ N*(N+1)/2`. Store:
  `s↦ N*(N+1)/2, i↦ N+1, n↦ N`.
- **B6 — `return s`.** `return` is `seqstrict`: `s` looks up to `N*(N+1)/2`.
  `(return)`: the rest of the callee body (`~> _`, here `.K`) is **discarded**,
  the frame is **popped**, the **caller store** (`result |-> _`) is **restored**,
  and the `Int` is delivered to `CONT`. Cooling fills the hole:
  `result = N*(N+1)/2`.
- **B7 — final assignment.** `(asgn)`: `result = (N*(N+1)/2) => .K`,
  `<store> result |-> _ => result |-> N*(N+1)/2`. The map-extensionality
  `[simplification]` reduces the post-store `#Equals` to VC3's scalar equality
  (Z3). Now `<k> = .K`, `<funcs> = sum|->...` (witnesses `?_`),
  `<stack> = .List` — exactly the post-state. ∎

`A ⊢ (SUM)` by `(Transitivity)` over B1–B7, with (LOOP) (proved in §5.2) used as
the loop lemma at B5.

### 5.4 Arithmetic verification conditions (product form)

Every numerator is a product of consecutive integers, hence even, so every
`/Int 2` is **exact** (VC-EXACT). Z3 discharges the linear facts; the exact-
halving `[simplification]` lemmas supply the facts where Z3 must equate two
distinct symbolic products under `/Int`.

| VC | Statement | Discharged by |
|---|---|---|
| **VC1** (loop step) | `I + cfA(I+1,N) = cfA(I,N)`, i.e. `I + (N+I+1)(N−I)/2 = (I+N)(N−I+1)/2` | exact-halving `[simplification]` + Z3 (after ×2 both sides expand to `N²+N−I²+I`) |
| **VC2** (loop exit) | `I = N+1 ⇒ cfA(I,N) = (2N+1)·0/2 = 0` | Z3 (zero factor) |
| **VC3** (init) | `cfA(1,N) = (1+N)·N/2 = N*(N+1)/2` | Z3 (after map-extensionality reduces the cell `#Equals` to this scalar) |
| side conditions | `N≥0 ⇒ 1≤N+1`; `I≤N ⇒ I+1≤N+1` | Z3 (linear) |
| map extensionality | `M[K<-V] #Equals M[K<-V'] ⇒ V #Equals V'` | `[simplification]`, then Z3 |

**Exact-halving note (flagged risk).** Lesson 1.22's *down*-counting loop needs
no simplifications because it never divides a *symbolic* product. Our
*up*-counting VC1 equates two distinct products under Euclidean `/Int`, where the
`×2` move is not linear without evenness — so the exact-halving lemmas are
**needed**, not held in reserve. The "Lesson 1.22 needs none ⇒ ours needs none"
inference is *not* assumed.

### 5.5 Soundness and residual risk

- **Scope: partial correctness.** If and when the loop terminates, the post-state
  holds. The circularity's guardedness gives coinductive *soundness*, not a
  well-founded variant; **termination is not separately proved** (no measure like
  `N − i`). Total correctness is out of scope.
- **`[all-path]` justification.** Both claims are `[all-path]` (stronger than the
  hand proof's one-path). This holds because mini Python is **deterministic** —
  at every redex exactly one rule or heating/cooling step applies; `(while-t)`/
  `(while-f)` are mutually exclusive on the guard; call/return/binding are
  syntax-directed. With determinism, all-path = one-path operationally. If the
  spec were generalized to non-deterministic constructs, this would need
  re-examination.
- **Trusted base.** (1) Adequacy of the mini Python semantics — including the
  `INDENT/DEDENT` indentation model and the `+=` value-faithfulness (exact only
  for side-effect-free simple-name targets), and the fragment restriction (no
  `if`/`None`/unbound-read, none exercised). (2) The reachability proof-system
  metatheory — in particular guarded-coinduction soundness of `(Circularity)` —
  and `kprove`'s soundness as its implementation. (3) The SMT/simplification
  oracle (Z3) plus the `[simplification]` lemmas, whose soundness (notably the
  evenness side condition on the second halving lemma) is the author's
  responsibility.
- **Openly flagged.** The K toolchain was **not executed in this environment**:
  the proof is machine-*unverified* here, though constructed to be checkable
  (every step cites a rule; it mirrors the proven Lesson 1.22). The independent
  assurance leg is the prior hand reachability proof
  ([sum-correctness-proof.md](sum-correctness-proof.md)), adversarially reviewed
  by 5 lenses (no critical/major defects) plus 20k randomized (LOOP) checks and
  `N = 0..11` end-to-end. Whether Z3 needs the halving lemmas should be confirmed
  empirically, and the FIX-2 parse should be confirmed with `kast`.

### 5.6 Reproducing the machine check

```sh
# 1. Compile the semantics (Haskell backend, required for proving)
kompile mini-python.k --backend haskell

# 2. (recommended) confirm both claim programs parse to a single AST
kast --backend haskell mini-python-spec.k

# 3. Discharge the claims; expected result:  #Top   (all claims proved)
kprove mini-python-spec.k
```
